# Yuxi-Know 离线交付部署说明

本文档用于把系统交付给对方运维（内网/断网环境）。

## 1. 交付材料清单

请至少交付以下文件：

1. 离线总包  
   `dist/yuxi-offline-remote-full.tar.gz`（当前约 4.0G）
2. 部署脚本  
   `scripts/offline_bundle.sh`
3. 编排文件  
   `docker-compose.remote.full.yml`
4. 环境变量模板/配置  
   `.env`（生产值，建议脱敏后单独加密传输）
5. 校验文件（建议）  
   `sha256sum.txt`（用于校验传输完整性）

## 2. 交付前在源机器打包

推荐一键打包（自动构建 `linux/amd64`、使用生产 tag、导出离线包并同步到 `delivery_materials`）：

```bash
bash scripts/package_delivery.sh
```

可选参数（环境变量方式）：

```bash
PLATFORM=linux/amd64 \
API_TAG=0.5.prod \
WEB_TAG=0.5.prod \
CRAWLER_TAG=0.1.prod \
BUNDLE_NAME=yuxi-offline-prod-20260306 \
bash scripts/package_delivery.sh
```

如果你只需要导出（不重新构建），可继续使用：

```bash
bash scripts/offline_bundle.sh export \
  --bundle-name yuxi-offline-remote-full \
  --compose-files "docker-compose.remote.full.yml" \
  --with-env yes \
  --target-platform linux/amd64
```

生成：

1. `dist/yuxi-offline-remote-full/`（目录）
2. `dist/yuxi-offline-remote-full.tar.gz`（最终交付包）

建议生成校验值：

```bash
cd dist
sha256sum yuxi-offline-remote-full.tar.gz > sha256sum.txt
```

## 3. 目标机器部署步骤

前置要求：

1. 已安装 Docker Engine（建议 v24+）
2. 已安装 Docker Compose（建议 v2.20+）
3. 目标机磁盘空间建议 >= 40G（解压 + 镜像导入 + 数据恢复）

### 3.1 准备目录

```bash
mkdir -p /opt/yuxi-know
cd /opt/yuxi-know
```

把以下文件拷贝到该目录：

1. `yuxi-offline-remote-full.tar.gz`
2. `scripts/offline_bundle.sh`
3. `docker-compose.remote.full.yml`
4. `.env`

### 3.2 校验包完整性（建议）

```bash
sha256sum -c sha256sum.txt
```

### 3.3 导入并启动

```bash
bash scripts/offline_bundle.sh import \
  --bundle ./yuxi-offline-remote-full.tar.gz \
  --compose-files "docker-compose.remote.full.yml" \
  --up yes
```

## 4. 启动后验收

### 4.1 容器状态

```bash
docker compose -f docker-compose.remote.full.yml ps
```

关键服务应为 `Up/healthy`：

1. `web-dev`
2. `api-dev`
3. `crawler-dev`
4. `crawler-worker-dev`
5. `postgres`
6. `milvus`
7. `graph`
8. `app-redis`
9. `prometheus`

### 4.2 健康检查

```bash
curl -sS http://127.0.0.1/api/system/health
curl -sS http://127.0.0.1/crawler-api/v1/health
```

说明：

1. 业务统一通过 Nginx 80 端口访问。
2. 爬虫接口走 `/crawler-api/*` 代理路径。

### 4.3 一键巡检脚本（推荐）

可直接执行仓库内脚本，自动检查容器状态、健康检查和网关代理：

```bash
bash scripts/check_services.sh --compose-files "docker-compose.remote.full.yml"
```

如需同时验证“登录后接口”是否可正常返回，请传管理员账号：

```bash
bash scripts/check_services.sh \
  --compose-files "docker-compose.remote.full.yml" \
  --admin-user "admin" \
  --admin-pass "Admin@123456"
```

当前交付默认管理员账号（用于验收测试）：

1. 用户名：`admin`
2. 密码：`Admin@123456`

注意：目标环境验收通过后，请立即修改该账号密码。

期望结果：

1. `PASS` 全通过
2. 汇总中 `FAIL=0`

脚本覆盖内容：

1. 容器运行态与健康态检查
2. 基础健康接口：`/api/system/health`、`/crawler-api/health`
3. 鉴权保护检查：未登录访问知识库接口返回 `401`
4. 登录后接口检查（提供管理员账号时）：
   `/api/auth/me`、`/api/departments/tree`、`/api/auth/users`、`/api/knowledge/databases`、`/crawler-api/tasks`、`/crawler-api/jobs`、`/crawler-api/logs`

### 4.4 关键功能抽检

1. 登录页面可访问
2. 知识库列表可加载
3. 知识库文档可下载/打开
4. 爬虫任务可创建并执行

## 5. 爬虫数据补丁恢复（可选）

当现场反馈“爬虫接口健康但 `tasks/jobs` 为空”时，按以下步骤恢复：

1. 将 `crawler-db-patch.tar.gz` 放到项目根目录，执行：

```bash
mkdir -p crawler_data
tar -xzf crawler-db-patch.tar.gz -C crawler_data
```

2. 执行修复脚本（会自动对齐数据库路径并重启 crawler 服务）：

```bash
bash fix_crawler_db_path.sh docker-compose.remote.full.yml
```

3. 验证：

```bash
curl "http://127.0.0.1/crawler-api/tasks?page=1&size=10"
curl "http://127.0.0.1/crawler-api/jobs?page=1&size=10"
```

交付目录中对应文件：

1. `patches/crawler-db-patch.tar.gz`
2. `fix_crawler_db_path.sh`

## 6. 运维常用命令

查看日志：

```bash
docker compose -f docker-compose.remote.full.yml logs -f web
docker compose -f docker-compose.remote.full.yml logs -f api
docker compose -f docker-compose.remote.full.yml logs -f crawler
docker compose -f docker-compose.remote.full.yml logs -f crawler-worker
```

重启：

```bash
docker compose -f docker-compose.remote.full.yml restart
```

停止：

```bash
docker compose -f docker-compose.remote.full.yml down
```

## 7. 常见问题

1. `502 Bad Gateway`  
   通常是 `api-dev` 仍在启动，先看 `docker compose ... logs -f api`。
2. `no space left on device`  
   磁盘不足，清理旧镜像/容器后重试。
3. 知识库文件打不开  
   先看 `api-dev` 日志，定位 `download` 相关报错。
4. 爬虫任务不执行  
   检查 `crawler-worker-dev` 是否运行、`app-redis` 是否正常。

## 8. 交付建议

1. `.env` 建议通过加密通道单独发送，不要和压缩包同渠道明文传输。
2. 首次上线完成后，建议由运维导出一份“可回滚快照”。
3. 明确变更流程：后续版本继续走“离线包 + 校验 + 导入”。
